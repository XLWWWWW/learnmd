## 第一章：为什么需要容器网络

容器本质上是运行在宿主机上的进程。

但是容器又不是普通进程。一个普通进程默认使用宿主机的文件系统、进程空间、网络栈；而容器通常被隔离在自己的命名空间中：

- 它看到的是自己的文件系统。
- 它看到的是自己的进程列表。
- 它看到的是自己的网卡、IP、路由表、端口。
- 它可以像一台“小机器”一样对外提供服务。

所以容器网络要解决的问题就是：

> 一个被隔离的进程，如何像普通机器一样接入网络？

更具体地说，容器网络主要解决四类问题：

1. 容器如何获得自己的 IP 地址。
2. 同一台宿主机上的容器如何互相通信。
3. 容器如何访问外部网络，例如访问公网、数据库、消息队列。
4. 外部用户如何访问容器内运行的服务。

如果是在 Kubernetes 中，还会继续扩展成：

1. 不同节点上的 Pod 如何互相通信。
2. Service 如何把流量转发到后端 Pod。
3. Pod IP、Service IP、Node IP、Ingress 域名之间是什么关系。
4. 网络策略如何限制访问。

### 1.1 从一个例子开始

假设启动一个 Nginx 容器：

```bash
docker run -d --name web nginx
```

这个容器内部会运行一个 Nginx 进程，监听 `80` 端口。

进入容器查看网络：

```bash
docker exec -it web sh
ip addr
ip route
```

你通常会看到类似信息：

```text
eth0@ifxx: <BROADCAST,MULTICAST,UP,LOWER_UP>
    inet 172.17.0.2/16

default via 172.17.0.1 dev eth0
```

这里有几个关键问题：

- `172.17.0.2` 是谁分配的？
- `eth0` 是真实网卡吗？
- `172.17.0.1` 是什么？
- 为什么容器能访问互联网？
- 为什么宿主机访问容器服务时经常要加 `-p 8080:80`？

容器网络的学习，就是把这些问题逐个讲清楚。

## 第二章：Linux 网络基础

容器网络不是凭空发明出来的，它主要建立在 Linux 网络能力之上。

理解容器网络之前，需要先理解几个基础概念。

### 2.1 网卡

网卡是网络通信的接口。

在 Linux 中，可以使用下面命令查看网卡：

```bash
ip link
ip addr
```

常见网卡：

- `eth0`：普通以太网网卡。
- `ens33`、`enp0s3`：不同命名规则下的物理或虚拟网卡。
- `lo`：本地回环网卡，地址通常是 `127.0.0.1`。
- `docker0`：Docker 默认创建的网桥。
- `vethxxx`：虚拟网卡的一端，常用于连接容器和宿主机。

### 2.2 IP 地址与网段

IP 地址用于标识网络中的一个主机或接口。

例如：

```text
192.168.1.10/24
```

其中：

- `192.168.1.10` 是 IP 地址。
- `/24` 是子网掩码长度。
- `/24` 等价于 `255.255.255.0`。
- 该网段范围大致是 `192.168.1.0` 到 `192.168.1.255`。

同一个二层网络中的主机可以通过 ARP 找到彼此的 MAC 地址，然后直接通信。

不同网段之间通信，需要经过路由器或网关。

### 2.3 路由表

路由表决定数据包下一跳去哪里。

查看路由：

```bash
ip route
```

示例：

```text
default via 192.168.1.1 dev eth0
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.10
```

含义：

- 访问 `192.168.1.0/24` 网段，直接从 `eth0` 发出去。
- 访问其他网段，走默认网关 `192.168.1.1`。

容器里也有自己的路由表。Docker 默认 bridge 网络下，容器的默认网关通常是 `docker0` 的地址。

### 2.4 端口

IP 用于定位主机，端口用于定位主机上的应用。

例如：

```text
192.168.1.10:80
```

表示访问 `192.168.1.10` 这台机器上的 `80` 端口。

容器内部服务监听的端口，默认只在容器网络命名空间内有效。如果想让宿主机或外部访问，一般需要端口映射：

```bash
docker run -d -p 8080:80 nginx
```

含义：

- 宿主机监听 `8080`。
- 流量转发到容器的 `80`。

### 2.5 NAT

NAT，全称 Network Address Translation，网络地址转换。

最常见的是源地址转换 SNAT：

```text
容器 172.17.0.2 访问公网 8.8.8.8

发出时：
172.17.0.2 -> 8.8.8.8

经过宿主机 NAT 后：
宿主机公网 IP -> 8.8.8.8
```

公网服务器回复给宿主机，宿主机再根据连接记录把响应转回容器。

Docker 容器能访问外网，通常就是依赖宿主机上的 NAT 规则。

还有目标地址转换 DNAT：

```text
外部访问宿主机 8080
宿主机把目标地址改成容器 172.17.0.2:80
```

Docker 的 `-p 8080:80` 背后就会涉及 DNAT。

### 2.6 iptables 与 nftables

Linux 中常用 `iptables` 或 `nftables` 管理防火墙和 NAT 规则。

Docker 会自动写入一些规则，用来实现：

- 容器访问外网的 SNAT/MASQUERADE。
- 宿主机端口到容器端口的 DNAT。
- 容器网络之间的隔离。

查看 NAT 表：

```bash
iptables -t nat -L -n -v
```

有些新系统底层使用 nftables，但仍然提供 `iptables` 兼容命令。

## 第三章：容器网络的底层积木

容器网络主要依赖这些 Linux 机制：

- Network Namespace
- veth pair
- Linux Bridge
- iptables/nftables
- 路由
- DNS

下面逐个说明。

### 3.1 Network Namespace

Network Namespace 是 Linux 的网络命名空间。

它可以让不同进程看到不同的网络环境。

每个网络命名空间都有自己独立的：

- 网卡列表
- IP 地址
- 路由表
- ARP 表
- iptables 规则
- 端口监听

这就是为什么两个容器都可以监听 `80` 端口而不冲突：

```bash
docker run -d --name web1 nginx
docker run -d --name web2 nginx
```

`web1` 内部可以监听 `0.0.0.0:80`，`web2` 内部也可以监听 `0.0.0.0:80`。

因为它们不是在同一个网络命名空间里监听端口。

### 3.2 手动创建 Network Namespace

可以不用 Docker，直接用 Linux 命令模拟容器网络。

创建命名空间：

```bash
ip netns add ns1
ip netns add ns2
```

查看：

```bash
ip netns list
```

在命名空间中执行命令：

```bash
ip netns exec ns1 ip addr
```

这时 `ns1` 里通常只有 `lo` 网卡。

启动回环网卡：

```bash
ip netns exec ns1 ip link set lo up
```

这说明网络命名空间确实拥有独立网络栈。

### 3.3 veth pair

veth pair 是一对虚拟网卡。

它的特点是：

> 从一端发出的包，会从另一端收到。

可以把它理解成一根虚拟网线。

创建 veth pair：

```bash
ip link add veth-a type veth peer name veth-b
```

把其中一端放入命名空间：

```bash
ip link set veth-b netns ns1
```

给两端配置 IP：

```bash
ip addr add 10.0.0.1/24 dev veth-a
ip link set veth-a up

ip netns exec ns1 ip addr add 10.0.0.2/24 dev veth-b
ip netns exec ns1 ip link set veth-b up
```

测试：

```bash
ping 10.0.0.2
ip netns exec ns1 ping 10.0.0.1
```

这就是容器与宿主机通信的基本模型。

### 3.4 Linux Bridge

如果只有两个网络命名空间，veth pair 就够了。

但是如果一台宿主机上有很多容器，不能每两个容器都拉一根线。

这时需要 Linux Bridge。

Bridge 可以理解成一个虚拟交换机：

```text
          Linux Bridge
        +--------------+
        |    br0       |
        +---+------+---+
            |      |
          veth1  veth2
            |      |
          ns1    ns2
```

创建 bridge：

```bash
ip link add br0 type bridge
ip addr add 10.0.0.1/24 dev br0
ip link set br0 up
```

把 veth 接到 bridge：

```bash
ip link set veth-a master br0
ip link set veth-a up
```

Docker 默认的 `docker0` 就是一个 Linux Bridge。

### 3.5 容器网络的经典结构

Docker 默认 bridge 网络大致如下：

```text
宿主机

              eth0
               |
         +-----+------+
         |   路由/NAT  |
         +-----+------+
               |
            docker0
        172.17.0.1/16
          /        \
       veth1      veth2
         |          |
      eth0        eth0
   172.17.0.2  172.17.0.3
     容器A       容器B
```

容器 A 访问容器 B：

```text
容器A eth0 -> veth1 -> docker0 -> veth2 -> 容器B eth0
```

容器访问外网：

```text
容器A eth0 -> veth1 -> docker0 -> 宿主机 NAT -> eth0 -> 外部网络
```

外部访问容器：

```text
外部请求 -> 宿主机 eth0:8080 -> DNAT -> 容器A 172.17.0.2:80
```

## 第四章：Docker 默认 bridge 网络

安装 Docker 后，通常会自动创建一个 `docker0` 网桥。

查看：

```bash
ip addr show docker0
docker network ls
docker network inspect bridge
```

默认网络列表通常类似：

```text
NETWORK ID     NAME      DRIVER    SCOPE
xxxx           bridge    bridge    local
xxxx           host      host      local
xxxx           none      null      local
```

其中：

- `bridge`：默认桥接网络。
- `host`：容器共享宿主机网络命名空间。
- `none`：容器只有 lo，没有普通网络连接。

### 4.1 默认 bridge 的创建流程

执行：

```bash
docker run -d --name c1 nginx
```

Docker 大致会做这些事情：

1. 创建容器的 Network Namespace。
2. 创建一对 veth pair。
3. 把一端放进容器命名空间，并命名为 `eth0`。
4. 把另一端留在宿主机，并接到 `docker0`。
5. 给容器 `eth0` 分配 IP，例如 `172.17.0.2/16`。
6. 设置容器默认路由，网关为 `172.17.0.1`。
7. 配置 NAT，使容器可以访问外网。

### 4.2 查看容器 IP

```bash
docker inspect c1
```

只查看 IP：

```bash
docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' c1
```

容器内查看：

```bash
docker exec c1 ip addr
docker exec c1 ip route
```

### 4.3 默认 bridge 的局限

默认 `bridge` 网络可以用，但生产中通常更推荐创建 user-defined bridge。

默认 bridge 的局限：

- 容器之间默认不能直接用容器名解析，老版本需要 `--link`。
- 网络隔离能力弱，不同业务都在默认 bridge 上容易混乱。
- 不便于按项目管理网段、别名、DNS。

创建自定义网络：

```bash
docker network create app-net
```

运行容器：

```bash
docker run -d --name app --network app-net nginx
docker run -it --rm --name client --network app-net busybox sh
```

在 `client` 中访问：

```bash
wget -qO- http://app
```

在 user-defined bridge 中，Docker 内置 DNS 可以解析容器名。

## 第五章：Docker 常见网络模式

Docker 常见网络模式：

- bridge
- host
- none
- container
- user-defined bridge
- overlay
- macvlan
- ipvlan

### 5.1 bridge 模式

bridge 是最常见模式。

使用：

```bash
docker run -d --name web --network bridge nginx
```

特点：

- 容器有独立网络命名空间。
- 容器有自己的 IP。
- 容器连接到宿主机的网桥。
- 出外网通常通过 NAT。
- 对外服务通常通过 `-p` 端口映射。

适合：

- 单机开发。
- 普通应用部署。
- Docker Compose 默认场景。

### 5.2 user-defined bridge 模式

创建：

```bash
docker network create mynet
docker run -d --name redis --network mynet redis
docker run -d --name api --network mynet my-api-image
```

`api` 可以直接访问：

```text
redis:6379
```

优点：

- 支持容器名 DNS 解析。
- 网络隔离更清晰。
- 可以自定义子网、网关。

创建自定义网段：

```bash
docker network create \
  --driver bridge \
  --subnet 10.10.0.0/24 \
  --gateway 10.10.0.1 \
  app-net
```

指定容器 IP：

```bash
docker run -d \
  --name web \
  --network app-net \
  --ip 10.10.0.10 \
  nginx
```

注意：固定 IP 在一些场景有用，但更推荐用服务名访问，而不是依赖固定 IP。

### 5.3 host 模式

使用：

```bash
docker run --network host nginx
```

host 模式下，容器不再拥有独立网络命名空间，而是直接使用宿主机网络。

特点：

- 没有容器独立 IP。
- 容器监听的端口就是宿主机端口。
- 网络性能损耗较低。
- 端口冲突风险更高。
- 隔离性较弱。

示例：

```bash
docker run --rm --network host nginx
```

如果 Nginx 在容器内监听 `80`，宿主机的 `80` 就会被占用。

适合：

- 对网络性能敏感的场景。
- 需要直接监听宿主机大量端口的场景。
- 某些网络诊断工具。

不适合：

- 多租户隔离。
- 同一宿主机运行多个相同端口服务。

### 5.4 none 模式

使用：

```bash
docker run -it --network none busybox sh
```

特点：

- 容器有自己的网络命名空间。
- 只有 `lo` 网卡。
- 没有普通网络连接。

适合：

- 需要完全手工配置网络。
- 安全隔离要求极高。
- 特殊实验。

### 5.5 container 模式

container 模式允许一个容器共享另一个容器的网络命名空间。

使用：

```bash
docker run -d --name app nginx
docker run -it --rm --network container:app busybox sh
```

第二个容器和 `app` 看到的是同一套网卡、IP、端口。

特点：

- 两个容器共享网络栈。
- 不能监听相同端口。
- 常用于 sidecar 模式。

Kubernetes 中同一个 Pod 里的多个容器，就是共享同一个网络命名空间。

### 5.6 overlay 模式

overlay 用于跨主机容器通信。

典型技术是 VXLAN。

在 Docker Swarm 中，可以创建 overlay 网络：

```bash
docker network create \
  --driver overlay \
  app-overlay
```

特点：

- 多台宿主机上的容器可以处在同一个逻辑网络中。
- 底层通过隧道封装跨主机转发。
- 需要集群控制面维护网络信息。

Kubernetes 中很多 CNI 插件也使用 overlay 思路，例如 Flannel VXLAN 模式。

### 5.7 macvlan 模式

macvlan 可以让容器像局域网中的一台独立机器。

创建：

```bash
docker network create -d macvlan \
  --subnet 192.168.1.0/24 \
  --gateway 192.168.1.1 \
  -o parent=eth0 \
  macvlan-net
```

运行：

```bash
docker run -d \
  --name web \
  --network macvlan-net \
  --ip 192.168.1.50 \
  nginx
```

特点：

- 容器拥有局域网 IP。
- 外部机器可以像访问普通主机一样访问容器。
- 宿主机与 macvlan 容器直接通信可能需要额外配置。
- 对底层交换机、云厂商网络环境有要求。

适合：

- 容器需要直接暴露在二层网络。
- 传统系统迁移，要求容器拥有固定局域网 IP。

### 5.8 ipvlan 模式

ipvlan 与 macvlan 类似，但多个容器可以共享父接口的 MAC 地址。

特点：

- 对交换机 MAC 表压力更小。
- 适合大规模容器网络。
- 配置和排查复杂度高于普通 bridge。

## 第六章：容器之间如何通信

容器通信可以分成三种情况：

1. 同一宿主机，同一网络。
2. 同一宿主机，不同网络。
3. 不同宿主机。

### 6.1 同一宿主机，同一 bridge 网络

创建网络：

```bash
docker network create demo-net
```

启动两个容器：

```bash
docker run -d --name web --network demo-net nginx
docker run -it --rm --name client --network demo-net busybox sh
```

在 `client` 中：

```bash
ping web
wget -qO- http://web
```

流量路径：

```text
client eth0
  -> client veth
  -> demo-net bridge
  -> web veth
  -> web eth0
```

DNS 解析：

```text
web -> web 容器 IP
```

这是 Docker 内置 DNS 实现的。

### 6.2 同一宿主机，不同 bridge 网络

创建两个网络：

```bash
docker network create net-a
docker network create net-b
```

启动：

```bash
docker run -d --name a --network net-a nginx
docker run -d --name b --network net-b nginx
```

默认情况下，`a` 和 `b` 不能直接通过容器名通信。

可以把一个容器接入多个网络：

```bash
docker network connect net-b a
```

此时 `a` 同时拥有 `net-a` 和 `net-b` 的网络接口。

查看：

```bash
docker inspect a
```

这种方式常用于：

- 一个反向代理容器连接多个业务网络。
- 一个 API 容器同时访问前端网络和数据库网络。

### 6.3 使用服务名而不是 IP

不要在应用配置中写死容器 IP：

```text
redis_host=172.18.0.2
```

更推荐：

```text
redis_host=redis
```

原因：

- 容器重建后 IP 可能变化。
- 服务名更稳定。
- Compose 和 Kubernetes 都鼓励用服务名发现服务。

### 6.4 不同宿主机容器通信

不同宿主机通信有多种方案：

- 端口映射：通过宿主机 IP 和端口访问。
- overlay 网络：容器 IP 直接跨主机通信。
- 路由方案：每个节点的容器网段可路由。
- Kubernetes CNI：由插件实现 Pod 间通信。

简单部署中常见方式：

```text
容器A -> 宿主机B IP:映射端口 -> 容器B
```

大规模集群中更常见：

```text
PodA IP -> 集群网络 -> PodB IP
```

## 第七章：容器如何访问外部网络

以默认 Docker bridge 为例。

容器访问公网：

```bash
docker run --rm busybox ping -c 3 8.8.8.8
```

如果能 ping 通，说明至少三件事正常：

- 容器有 IP。
- 容器有默认路由。
- 宿主机 NAT 和外部网络正常。

### 7.1 出口流量路径

```text
容器 172.17.0.2
  -> eth0
  -> veth pair
  -> docker0 172.17.0.1
  -> 宿主机路由
  -> MASQUERADE/SNAT
  -> 宿主机 eth0
  -> 外部网络
```

### 7.2 MASQUERADE

Docker 通常会添加类似规则：

```text
-A POSTROUTING -s 172.17.0.0/16 ! -o docker0 -j MASQUERADE
```

含义：

- 源地址来自 `172.17.0.0/16`。
- 出口不是 `docker0`。
- 做 MASQUERADE，把源地址改成宿主机出口地址。

查看：

```bash
iptables -t nat -S
```

### 7.3 DNS 问题

容器能 ping IP，但不能解析域名：

```bash
docker run --rm busybox ping -c 3 8.8.8.8
docker run --rm busybox nslookup example.com
```

如果前者正常，后者失败，通常是 DNS 问题。

查看容器 DNS：

```bash
docker run --rm busybox cat /etc/resolv.conf
```

运行时指定 DNS：

```bash
docker run --rm --dns 8.8.8.8 busybox nslookup example.com
```

Docker daemon 级别配置可写在：

```text
/etc/docker/daemon.json
```

示例：

```json
{
  "dns": ["8.8.8.8", "1.1.1.1"]
}
```

修改后重启 Docker：

```bash
systemctl restart docker
```

## 第八章：外部如何访问容器

容器内部监听端口，并不等于外部可以访问。

### 8.1 不发布端口

```bash
docker run -d --name web nginx
```

查看：

```bash
docker ps
```

可能看到：

```text
80/tcp
```

这只表示镜像声明或容器内部使用了 `80` 端口，不表示宿主机已经监听。

宿主机访问容器 IP 可能可行：

```bash
curl http://172.17.0.2
```

但外部机器通常不能直接访问这个私有容器 IP。

### 8.2 使用 -p 发布端口

```bash
docker run -d --name web -p 8080:80 nginx
```

含义：

```text
宿主机 8080 -> 容器 80
```

访问：

```bash
curl http://localhost:8080
curl http://宿主机IP:8080
```

查看端口映射：

```bash
docker port web
```

### 8.3 绑定指定地址

只允许本机访问：

```bash
docker run -d -p 127.0.0.1:8080:80 nginx
```

允许所有网卡访问：

```bash
docker run -d -p 0.0.0.0:8080:80 nginx
```

实际部署时要注意：

- `0.0.0.0` 会暴露到所有可达网络。
- 云服务器还需要安全组放行端口。
- 本机防火墙也可能阻断访问。

### 8.4 EXPOSE 与 -p 的区别

Dockerfile 中：

```dockerfile
EXPOSE 80
```

`EXPOSE` 只是声明容器倾向使用的端口，不会自动发布端口。

真正发布端口的是：

```bash
docker run -p 8080:80 nginx
```

或者随机发布：

```bash
docker run -P nginx
```

`-P` 会把镜像声明的端口随机映射到宿主机高位端口。

### 8.5 端口映射的底层原理

端口映射一般由两部分实现：

- iptables/nftables DNAT 规则。
- docker-proxy，部分场景下辅助处理用户态转发。

流量路径：

```text
外部用户
  -> 宿主机 IP:8080
  -> DNAT
  -> 容器 IP:80
  -> Nginx
```

查看监听：

```bash
ss -lntp
```

查看 NAT 规则：

```bash
iptables -t nat -L -n -v
```

## 第九章：Docker Compose 网络

Docker Compose 默认会为一个项目创建独立网络。

示例 `compose.yaml`：

```yaml
services:
  web:
    image: nginx
    ports:
      - "8080:80"

  app:
    image: busybox
    command: sh -c "while true; do sleep 3600; done"
```

启动：

```bash
docker compose up -d
```

Compose 通常会创建类似网络：

```text
目录名_default
```

查看：

```bash
docker network ls
docker compose ps
```

### 9.1 服务名 DNS

在 `app` 容器中访问 `web`：

```bash
docker compose exec app sh
wget -qO- http://web
```

在 Compose 网络中：

```text
web -> web 服务容器 IP
```

如果 `web` 扩容成多个副本，服务名可能解析出多个 IP，客户端或 DNS 轮询会参与负载分发。

### 9.2 自定义网络

```yaml
services:
  web:
    image: nginx
    networks:
      - frontend

  api:
    image: my-api
    networks:
      - frontend
      - backend

  db:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: example
    networks:
      - backend

networks:
  frontend:
  backend:
```

含义：

- `web` 只能访问 `frontend` 中的服务。
- `db` 只在 `backend` 中。
- `api` 同时连接 `frontend` 和 `backend`，可以作为中间层。

这种网络划分比所有容器都放在一个网络中更清晰。

### 9.3 ports 与 expose

```yaml
services:
  web:
    image: nginx
    ports:
      - "8080:80"
```

`ports` 会发布到宿主机。

```yaml
services:
  api:
    image: my-api
    expose:
      - "8080"
```

`expose` 只表达容器内部网络可访问端口，不发布到宿主机。

## 第十章：跨主机容器网络

单机网络中，bridge 足够使用。

但集群中，容器可能分布在不同宿主机：

```text
Node1: PodA 10.244.1.10
Node2: PodB 10.244.2.20
```

问题是：

> PodA 如何直接访问 PodB？

跨主机容器网络通常有两类方案：

1. Overlay 隧道网络。
2. Underlay 路由网络。

### 10.1 Overlay 网络

Overlay 的思想是：

> 在已有宿主机网络之上，再构建一层虚拟网络。

常见封装技术：

- VXLAN
- Geneve
- IP-in-IP

VXLAN 示例路径：

```text
PodA
  -> Node1 cni0
  -> 封装成 VXLAN 包
  -> Node1 eth0
  -> 物理网络
  -> Node2 eth0
  -> 解封装
  -> Node2 cni0
  -> PodB
```

优点：

- 对底层网络要求低。
- 不需要物理网络知道每个 Pod 网段。
- 容易部署。

缺点：

- 有封装开销。
- MTU 问题更常见。
- 排查比普通路由复杂。

### 10.2 Underlay 路由网络

Underlay 的思想是：

> 让底层网络直接知道容器或 Pod 网段如何转发。

例如：

```text
Node1 负责 10.244.1.0/24
Node2 负责 10.244.2.0/24

路由：
10.244.1.0/24 -> Node1 IP
10.244.2.0/24 -> Node2 IP
```

可以通过静态路由，也可以通过 BGP 动态发布路由。

常见插件：

- Calico BGP 模式。
- Cilium native routing 模式。

优点：

- 性能好。
- 路径清晰。
- 不需要隧道封装。

缺点：

- 对网络环境要求更高。
- 云环境中可能受路由表、安全组、反欺骗策略限制。

### 10.3 MTU 问题

Overlay 网络会增加额外包头。

例如原始以太网 MTU 是 `1500`，VXLAN 封装会占用一部分空间。

如果没有正确调整 MTU，可能出现：

- 小包正常，大包失败。
- ping 正常，HTTP 下载卡住。
- TLS 握手异常。

排查：

```bash
ping -M do -s 1472 目标IP
ping -M do -s 1400 目标IP
```

说明：

- `-M do` 表示禁止分片。
- `-s` 指定 ICMP payload 大小。
- 找到可通过的最大包大小，再判断 MTU 是否匹配。

## 第十一章：Kubernetes 网络模型

Kubernetes 对网络有明确模型。

核心原则：

1. 每个 Pod 都有独立 IP。
2. 所有 Pod 之间默认可以直接通信，不需要 NAT。
3. Node 可以和所有 Pod 通信。
4. Pod 内部容器共享网络命名空间。

### 11.1 Pod 网络

一个 Pod 中可以有多个容器。

这些容器共享：

- 同一个 IP。
- 同一个端口空间。
- 同一个路由表。
- 同一个网络命名空间。

例如一个 Pod 有两个容器：

```text
Pod IP: 10.244.1.10

Container A: app，监听 8080
Container B: sidecar，监听 15000
```

它们可以通过 `localhost` 互相访问：

```text
Container B -> http://127.0.0.1:8080
```

但不能都监听同一个端口。

### 11.2 Pod 到 Pod

同节点 Pod 通信：

```text
PodA eth0
  -> veth
  -> cni0 或其他 bridge
  -> veth
  -> PodB eth0
```

跨节点 Pod 通信：

```text
PodA
  -> Node1
  -> CNI 网络
  -> Node2
  -> PodB
```

具体路径由 CNI 插件决定。

### 11.3 Pod 到外部网络

Pod 访问外部网络时，通常会经过节点出口。

有些集群会对 Pod 出口做 SNAT：

```text
Pod IP -> Node IP -> 外部网络
```

是否 SNAT 取决于 CNI 配置、云网络和集群策略。

### 11.4 外部访问 Pod

不要直接依赖 Pod IP 对外提供服务。

原因：

- Pod 会重建。
- Pod IP 会变化。
- Pod 数量会扩缩容。
- Pod 可能漂移到不同 Node。

Kubernetes 使用 Service 暴露稳定访问入口。

## 第十二章：CNI 插件原理

CNI，全称 Container Network Interface。

它是一套规范，用来让容器运行时调用网络插件完成网络配置。

Kubernetes 创建 Pod 时，需要有人完成这些工作：

- 创建或使用 Pod 的 network namespace。
- 创建 veth pair。
- 把一端放入 Pod。
- 配置 Pod IP。
- 配置路由。
- 配置跨节点通信规则。
- 在删除 Pod 时清理网络资源。

这些工作由 CNI 插件完成。

### 12.1 CNI 的调用时机

简化流程：

```text
kubelet
  -> container runtime
  -> CNI plugin ADD
  -> 配置 Pod 网络
  -> 启动 Pod 容器
```

删除时：

```text
kubelet
  -> container runtime
  -> CNI plugin DEL
  -> 清理 Pod 网络
```

### 12.2 CNI 配置文件

CNI 配置通常位于：

```text
/etc/cni/net.d/
```

CNI 插件二进制通常位于：

```text
/opt/cni/bin/
```

示例配置可能类似：

```json
{
  "cniVersion": "0.3.1",
  "name": "mynet",
  "type": "bridge",
  "bridge": "cni0",
  "isGateway": true,
  "ipMasq": true,
  "ipam": {
    "type": "host-local",
    "subnet": "10.244.0.0/16"
  }
}
```

字段含义：

- `type`：使用哪个 CNI 插件。
- `bridge`：网桥名称。
- `isGateway`：是否把 bridge 作为网关。
- `ipMasq`：是否做 IP 伪装。
- `ipam`：IP 地址管理配置。

### 12.3 IPAM

IPAM 是 IP Address Management。

它负责分配和回收 IP。

常见方式：

- 每个节点一个 PodCIDR。
- 从本地文件记录已分配 IP。
- 从集群存储中分配 IP。
- 由云厂商 VPC 分配 IP。

例如 Kubernetes 节点：

```text
Node1 PodCIDR: 10.244.1.0/24
Node2 PodCIDR: 10.244.2.0/24
```

Node1 上的 Pod 从 `10.244.1.0/24` 中分配 IP。

Node2 上的 Pod 从 `10.244.2.0/24` 中分配 IP。

### 12.4 常见 CNI 插件

常见插件：

- Flannel：简单易用，常见 VXLAN 模式。
- Calico：支持 BGP、网络策略，生产常见。
- Cilium：基于 eBPF，能力强，支持高级观测和策略。
- Weave Net：早期常用 overlay 方案。
- Canal：Flannel + Calico policy 的组合思路。

选择插件时主要看：

- 网络模式：overlay 还是路由。
- 是否需要 NetworkPolicy。
- 是否运行在云厂商环境。
- 性能要求。
- 运维复杂度。

### 12.5 Flannel VXLAN 简化原理

Flannel 常见模式是 VXLAN。

每个节点分配一个子网：

```text
Node1: 10.244.1.0/24
Node2: 10.244.2.0/24
```

Node1 上的 Pod 访问 Node2 上的 Pod：

```text
10.244.1.10 -> 10.244.2.20
```

Node1 查询到 `10.244.2.0/24` 在 Node2，于是把包封装成 VXLAN，通过宿主机网络发到 Node2。Node2 解封装后转给目标 Pod。

### 12.6 Calico 简化原理

Calico 更偏路由模型。

它可以通过 BGP 分发路由：

```text
10.244.1.0/24 via Node1
10.244.2.0/24 via Node2
```

这样跨节点 Pod 通信不一定需要 overlay 封装。

Calico 也常用于 NetworkPolicy：

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-to-db
spec:
  podSelector:
    matchLabels:
      app: db
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: api
      ports:
        - protocol: TCP
          port: 3306
```

含义：

- 选择 `app=db` 的 Pod。
- 只允许 `app=api` 的 Pod 访问它的 `3306`。

### 12.7 Cilium 简化原理

Cilium 基于 eBPF 实现网络、负载均衡、安全策略和观测。

eBPF 可以在内核中的特定 hook 点执行受限制的小程序。

Cilium 常见能力：

- Pod 网络。
- Service 负载均衡。
- NetworkPolicy。
- L7 策略。
- Hubble 网络观测。
- kube-proxy 替代。

可以简单理解为：

> Cilium 把很多原本依赖 iptables 的网络转发和策略能力，搬到 eBPF 数据路径中实现。

## 第十三章：Service、Ingress 与负载均衡

Kubernetes 中 Pod IP 不稳定，因此需要 Service。

### 13.1 Service 是什么

Service 是一组 Pod 的稳定访问入口。

示例：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080
```

含义：

- Service 名称：`web`
- Service 端口：`80`
- 后端 Pod 端口：`8080`
- 后端选择器：`app=web`

访问：

```text
http://web:80
```

### 13.2 Endpoint 与 EndpointSlice

Service 本身不运行应用。

真正运行应用的是 Pod。

Service 通过 selector 找到 Pod，然后生成 Endpoint 或 EndpointSlice。

查看：

```bash
kubectl get svc
kubectl get endpoints
kubectl get endpointslice
```

如果 Service 访问失败，应该检查：

- Service selector 是否匹配 Pod label。
- Pod 是否 Ready。
- targetPort 是否正确。
- EndpointSlice 是否有后端地址。

### 13.3 ClusterIP

ClusterIP 是默认类型。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080
```

特点：

- 只能在集群内部访问。
- 有一个虚拟 IP。
- kube-proxy 或 CNI 数据路径负责转发到后端 Pod。

访问路径：

```text
PodA -> Service ClusterIP:80 -> PodB:8080
```

### 13.4 NodePort

NodePort 会在每个节点上开放一个端口。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  type: NodePort
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080
```

访问：

```text
http://任意节点IP:30080
```

流量路径：

```text
外部用户 -> NodeIP:30080 -> Service -> Pod
```

适合：

- 简单暴露服务。
- 测试环境。
- 负载均衡器后端。

不足：

- 端口范围有限。
- 暴露方式不够灵活。
- 生产中通常配合 LoadBalancer 或 Ingress。

### 13.5 LoadBalancer

在云环境中，Service 类型可以是 LoadBalancer：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080
```

云厂商会创建一个外部负载均衡器，把流量转发到集群节点或 Pod。

本地环境如果没有云厂商控制器，可以使用 MetalLB 提供类似能力。

### 13.6 Ingress

Ingress 用于 HTTP/HTTPS 七层入口。

示例：

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web
spec:
  rules:
    - host: web.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web
                port:
                  number: 80
```

访问路径：

```text
用户浏览器
  -> web.example.com
  -> Ingress Controller
  -> Service web
  -> Pod
```

注意：

- Ingress 只是规则。
- 真正接收流量的是 Ingress Controller。
- 常见实现有 Nginx Ingress Controller、Traefik、HAProxy、云厂商 Ingress 等。

### 13.7 kube-proxy

kube-proxy 负责实现 Service 转发。

常见模式：

- iptables
- IPVS
- userspace，较老

iptables 模式：

```text
访问 ClusterIP
  -> iptables 规则
  -> 选择一个 Pod 后端
  -> DNAT 到 Pod IP
```

IPVS 模式：

```text
访问 ClusterIP
  -> IPVS 虚拟服务
  -> 负载均衡到 Pod
```

有些 CNI，例如 Cilium，可以替代 kube-proxy，用 eBPF 实现 Service 转发。

## 第十四章：DNS 与服务发现

容器网络不能只看 IP，还要看名字解析。

### 14.1 Docker DNS

在 user-defined bridge 网络中，容器可以用容器名互相访问：

```bash
docker network create demo
docker run -d --name redis --network demo redis
docker run -it --rm --network demo busybox sh
```

容器内：

```bash
nslookup redis
```

Docker 内置 DNS 会返回 `redis` 容器在该网络中的 IP。

### 14.2 Kubernetes DNS

Kubernetes 通常使用 CoreDNS。

Service 会自动获得 DNS 名称：

```text
service-name.namespace.svc.cluster.local
```

例如：

```text
web.default.svc.cluster.local
```

同 namespace 内可以简写：

```text
web
```

不同 namespace：

```text
web.prod
web.prod.svc
web.prod.svc.cluster.local
```

### 14.3 Pod 中的 resolv.conf

查看：

```bash
kubectl exec -it pod-name -- cat /etc/resolv.conf
```

可能看到：

```text
nameserver 10.96.0.10
search default.svc.cluster.local svc.cluster.local cluster.local
options ndots:5
```

含义：

- `nameserver` 指向 CoreDNS 的 Service IP。
- `search` 决定短名称如何补全。
- `ndots:5` 会影响域名查询顺序。

### 14.4 Headless Service

普通 Service 返回 ClusterIP。

Headless Service 没有 ClusterIP：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: db
spec:
  clusterIP: None
  selector:
    app: db
  ports:
    - port: 5432
```

DNS 查询会直接返回后端 Pod IP。

适合：

- StatefulSet。
- 数据库集群。
- 客户端需要感知每个实例的场景。

## 第十五章：容器网络配置

前面讲的是容器网络原理，这一章重点讲配置。

容器网络配置可以分成三层：

1. 宿主机层配置：网卡、路由、转发、防火墙、DNS。
2. Docker 层配置：网络驱动、子网、网关、端口映射、DNS。
3. Kubernetes 层配置：Pod 网段、Service 网段、CNI、Service、Ingress、NetworkPolicy。

学习网络配置时，不要只背命令，而要理解每个配置改变的是哪一层。

### 15.1 宿主机网络配置

容器网络最终还是跑在宿主机网络之上。

如果宿主机网络不通，容器网络通常也不会正常。

查看宿主机 IP：

```bash
ip addr
```

查看路由：

```bash
ip route
```

查看 DNS：

```bash
cat /etc/resolv.conf
```

查看监听端口：

```bash
ss -lntp
```

查看防火墙规则：

```bash
iptables -L -n -v
iptables -t nat -L -n -v
```

容器访问外网依赖宿主机转发能力。

查看 IPv4 转发：

```bash
sysctl net.ipv4.ip_forward
```

如果结果是：

```text
net.ipv4.ip_forward = 0
```

表示内核没有开启 IPv4 转发。

临时开启：

```bash
sysctl -w net.ipv4.ip_forward=1
```

永久配置可以写入：

```text
/etc/sysctl.conf
```

或：

```text
/etc/sysctl.d/*.conf
```

示例：

```text
net.ipv4.ip_forward = 1
```

加载：

```bash
sysctl -p
```

### 15.2 Docker daemon 网络配置

Docker daemon 的全局配置文件通常是：

```text
/etc/docker/daemon.json
```

常见网络相关配置：

```json
{
  "bip": "172.18.0.1/16",
  "fixed-cidr": "172.18.0.0/16",
  "dns": ["8.8.8.8", "1.1.1.1"],
  "default-address-pools": [
    {
      "base": "172.30.0.0/16",
      "size": 24
    }
  ]
}
```

字段说明：

- `bip`：默认 `docker0` 网桥地址。
- `fixed-cidr`：默认 bridge 网络可分配给容器的地址范围。
- `dns`：容器默认使用的 DNS。
- `default-address-pools`：新建自定义 bridge 网络时默认从哪些地址池分配网段。

修改后重启 Docker：

```bash
systemctl restart docker
```

注意：修改 `bip`、地址池、bridge 配置可能影响已有容器网络，生产环境需要提前评估。

### 15.3 创建自定义 bridge 网络

实际项目中更推荐使用自定义 bridge 网络，而不是所有容器都挂到默认 `bridge`。

创建普通自定义网络：

```bash
docker network create app-net
```

创建指定网段的网络：

```bash
docker network create \
  --driver bridge \
  --subnet 10.20.0.0/24 \
  --gateway 10.20.0.1 \
  app-net
```

运行容器：

```bash
docker run -d --name web --network app-net nginx
```

查看网络：

```bash
docker network inspect app-net
```

在自定义 bridge 中，容器名可以作为 DNS 名称：

```bash
docker run -it --rm --network app-net busybox sh
```

容器内：

```bash
nslookup web
wget -qO- http://web
```

### 15.4 给容器指定网络参数

运行容器时可以指定 IP：

```bash
docker run -d \
  --name web \
  --network app-net \
  --ip 10.20.0.10 \
  nginx
```

指定 DNS：

```bash
docker run --rm \
  --dns 8.8.8.8 \
  busybox nslookup example.com
```

指定主机名：

```bash
docker run -it --rm \
  --hostname app01 \
  busybox sh
```

添加 hosts 记录：

```bash
docker run --rm \
  --add-host db.local:10.20.0.20 \
  busybox cat /etc/hosts
```

这些配置最终会影响容器中的：

- `/etc/hosts`
- `/etc/hostname`
- `/etc/resolv.conf`
- 网卡 IP
- 路由表

### 15.5 端口发布配置

发布单个端口：

```bash
docker run -d -p 8080:80 nginx
```

绑定本地回环地址：

```bash
docker run -d -p 127.0.0.1:8080:80 nginx
```

绑定所有地址：

```bash
docker run -d -p 0.0.0.0:8080:80 nginx
```

发布 UDP 端口：

```bash
docker run -d -p 5353:53/udp dns-image
```

随机端口发布：

```bash
docker run -d -P nginx
```

查看发布结果：

```bash
docker ps
docker port 容器名
```

端口发布配置要注意三个点：

1. 容器内应用必须监听正确端口。
2. 应用最好监听 `0.0.0.0`，不要只监听 `127.0.0.1`。
3. 宿主机防火墙、云安全组也要允许该端口。

### 15.6 Docker Compose 网络配置

Compose 中可以声明多个网络：

```yaml
services:
  nginx:
    image: nginx
    ports:
      - "8080:80"
    networks:
      - frontend

  api:
    image: my-api
    networks:
      - frontend
      - backend

  mysql:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: example
    networks:
      - backend

networks:
  frontend:
    driver: bridge
  backend:
    driver: bridge
```

这个配置表达的是：

- `nginx` 和 `api` 在 `frontend` 网络中，可以互相访问。
- `api` 和 `mysql` 在 `backend` 网络中，可以互相访问。
- `nginx` 不能直接访问 `mysql`。

指定网段：

```yaml
networks:
  backend:
    driver: bridge
    ipam:
      config:
        - subnet: 10.30.0.0/24
          gateway: 10.30.0.1
```

设置服务别名：

```yaml
services:
  mysql:
    image: mysql:8
    networks:
      backend:
        aliases:
          - db
```

此时同网络中的其他容器可以访问：

```text
mysql:3306
db:3306
```

### 15.7 Kubernetes Pod 网络配置

Pod 中最常见的网络配置包括：

- `containerPort`
- `hostNetwork`
- `dnsPolicy`
- `dnsConfig`
- `hostAliases`

示例：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: network-demo
spec:
  containers:
    - name: nginx
      image: nginx
      ports:
        - containerPort: 80
```

`containerPort` 只是声明容器端口，方便工具和其他资源引用，不等于自动创建 Service，也不等于对外暴露。

使用宿主机网络：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: host-network-demo
spec:
  hostNetwork: true
  containers:
    - name: nginx
      image: nginx
```

`hostNetwork: true` 表示 Pod 直接共享节点网络命名空间。

特点：

- Pod 没有普通独立 Pod IP。
- 容器监听端口会直接占用节点端口。
- DNS 策略通常需要配合 `ClusterFirstWithHostNet`。

示例：

```yaml
spec:
  hostNetwork: true
  dnsPolicy: ClusterFirstWithHostNet
```

### 15.8 Kubernetes DNS 配置

默认 Pod DNS 策略通常是：

```yaml
dnsPolicy: ClusterFirst
```

含义：

- 集群内域名走 CoreDNS。
- 外部域名再按上游 DNS 解析。

自定义 DNS：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: dns-demo
spec:
  dnsPolicy: None
  dnsConfig:
    nameservers:
      - 8.8.8.8
    searches:
      - example.local
    options:
      - name: ndots
        value: "2"
  containers:
    - name: busybox
      image: busybox:1.36
      command: ["sleep", "3600"]
```

注意：业务 Pod 通常不建议随意改成外部 DNS，否则可能解析不了集群内 Service。

### 15.9 Kubernetes Service 配置

Service 配置的关键是 selector 和端口。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api
spec:
  selector:
    app: api
  ports:
    - name: http
      port: 80
      targetPort: 8080
```

含义：

- `port: 80`：Service 暴露的端口。
- `targetPort: 8080`：后端 Pod 的端口。
- `selector: app=api`：选择哪些 Pod 作为后端。

最容易出错的是：

- selector 和 Pod label 对不上。
- targetPort 写错。
- Pod 应用没有监听 `0.0.0.0:8080`。

### 15.10 NetworkPolicy 配置

NetworkPolicy 用于限制 Pod 间访问。

示例：只允许 `app=api` 访问 `app=db` 的 `3306`。

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-allow-api
spec:
  podSelector:
    matchLabels:
      app: db
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: api
      ports:
        - protocol: TCP
          port: 3306
```

注意：

- NetworkPolicy 是否生效，取决于 CNI 是否支持。
- 一旦 Pod 被 Ingress 策略选中，未显式允许的入站流量通常会被拒绝。
- 如果限制出站流量，要记得允许 DNS。

允许访问 CoreDNS 的示例思路：

```yaml
egress:
  - to:
      - namespaceSelector:
          matchLabels:
            kubernetes.io/metadata.name: kube-system
    ports:
      - protocol: UDP
        port: 53
      - protocol: TCP
        port: 53
```

### 15.11 网络配置的基本原则

容器网络配置建议遵循这些原则：

1. 同一业务使用独立网络，不要全部放进默认 bridge。
2. 服务之间优先使用服务名访问，不要依赖容器 IP。
3. 对外暴露端口时明确绑定地址，避免无意暴露。
4. 应用监听 `0.0.0.0`，不要只监听 `127.0.0.1`。
5. Compose 中按前端、后端、数据库拆分网络。
6. Kubernetes 中用 Service 暴露稳定入口，不直接依赖 Pod IP。
7. 使用 NetworkPolicy 时先画清楚访问关系，再写规则。
8. 改 Docker daemon 或 CNI 配置前先评估已有容器和 Pod 的影响。

## 第十六章：容器进程管理

容器网络和进程管理关系非常紧密。

原因很简单：

> 网络端口最终是进程监听的，容器生命周期也由主进程决定。

如果进程没有启动，端口不会监听。

如果进程只监听 `127.0.0.1`，其他容器无法通过容器 IP 访问。

如果主进程退出，容器也会退出。

### 16.1 容器的本质是进程

运行一个容器：

```bash
docker run -d --name web nginx
```

在宿主机上查看进程：

```bash
ps aux | grep nginx
```

容器里的 Nginx 进程，本质上仍然是宿主机内核管理的普通进程，只是被 namespace 和 cgroup 隔离、限制。

容器主要涉及这些隔离：

- PID Namespace：进程编号隔离。
- Network Namespace：网络栈隔离。
- Mount Namespace：文件系统挂载隔离。
- UTS Namespace：主机名隔离。
- IPC Namespace：进程间通信隔离。
- User Namespace：用户映射隔离。
- cgroup：资源限制和统计。

### 16.2 PID 1 的特殊性

容器启动后，容器内的主进程通常是 PID 1。

查看：

```bash
docker exec web ps -ef
```

你可能看到：

```text
UID   PID  PPID  CMD
root    1     0  nginx: master process nginx -g daemon off;
```

PID 1 在 Linux 中有特殊职责：

- 接收容器停止时发送的信号。
- 回收子进程，避免僵尸进程。
- 决定容器是否继续运行。

如果 PID 1 不正确处理信号，可能导致：

- `docker stop` 很慢。
- 应用不能优雅退出。
- 请求还没处理完就被强制杀死。
- 子进程变成僵尸进程。

### 16.3 前台进程与后台进程

容器要求主进程在前台运行。

错误示例：

```dockerfile
CMD service nginx start
```

这类命令可能启动后台服务后立即退出，导致容器退出。

正确示例：

```dockerfile
CMD ["nginx", "-g", "daemon off;"]
```

对 Java 应用：

```dockerfile
CMD ["java", "-jar", "app.jar"]
```

对 Node.js 应用：

```dockerfile
CMD ["node", "server.js"]
```

核心原则：

> 容器的主进程就是应用本身，不要把应用藏在后台。

### 16.4 CMD 与 ENTRYPOINT

Dockerfile 中常见两个指令：

- `CMD`
- `ENTRYPOINT`

`CMD` 更像默认参数或默认命令。

```dockerfile
CMD ["nginx", "-g", "daemon off;"]
```

运行时可以覆盖：

```bash
docker run nginx echo hello
```

`ENTRYPOINT` 更像固定入口。

```dockerfile
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
```

配合 `CMD`：

```dockerfile
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
CMD ["--server.port=8080"]
```

运行时追加或覆盖参数：

```bash
docker run app-image --server.port=9090
```

### 16.5 shell 格式与 exec 格式

Dockerfile 中有两种写法。

shell 格式：

```dockerfile
CMD java -jar app.jar
```

实际会变成：

```text
/bin/sh -c "java -jar app.jar"
```

exec 格式：

```dockerfile
CMD ["java", "-jar", "app.jar"]
```

推荐使用 exec 格式。

原因：

- 应用进程更容易直接成为 PID 1。
- 信号传递更清晰。
- 参数处理更可靠。

shell 格式可能导致信号先发给 shell，而不是应用本身。

### 16.6 docker stop 的过程

执行：

```bash
docker stop web
```

默认流程：

```text
Docker -> 向容器 PID 1 发送 SIGTERM
等待一段时间
如果还没退出 -> 发送 SIGKILL
```

可以指定等待时间：

```bash
docker stop -t 30 web
```

含义：

- 给应用 30 秒优雅退出。
- 超时后强制杀死。

应用应该正确处理 SIGTERM：

- 停止接收新请求。
- 等待正在处理的请求完成。
- 关闭数据库连接。
- 刷新日志和缓冲区。
- 正常退出进程。

### 16.7 docker kill、pause、restart

强制发送信号：

```bash
docker kill web
```

发送指定信号：

```bash
docker kill --signal=HUP web
```

暂停容器进程：

```bash
docker pause web
```

恢复：

```bash
docker unpause web
```

重启：

```bash
docker restart web
```

区别：

- `stop`：优雅停止，先 SIGTERM 后 SIGKILL。
- `kill`：默认直接 SIGKILL。
- `pause`：冻结进程，不释放端口和资源状态。
- `restart`：停止后再启动。

### 16.8 查看容器进程

查看容器主进程：

```bash
docker top web
```

进入容器查看：

```bash
docker exec -it web sh
ps -ef
```

查看容器详细状态：

```bash
docker inspect web
```

查看 PID：

```bash
docker inspect -f '{{.State.Pid}}' web
```

拿到宿主机 PID 后，可以进入对应 namespace 排查：

```bash
nsenter -t 宿主机PID -n ip addr
nsenter -t 宿主机PID -n ss -lntp
```

这对网络排查非常有用，因为可以直接进入容器网络命名空间查看监听和路由。

### 16.9 容器日志与进程输出

容器最佳实践是把日志输出到标准输出和标准错误：

```text
stdout
stderr
```

Docker 查看日志：

```bash
docker logs web
docker logs -f web
docker logs --tail 100 web
```

不要默认把日志只写到容器内部文件中。

原因：

- 容器重建后本地文件可能丢失。
- 日志采集系统通常从 stdout/stderr 收集。
- 排查容器启动失败时 `docker logs` 最直接。

### 16.10 资源限制与进程管理

容器可以通过 cgroup 限制资源。

限制内存：

```bash
docker run -d --name app --memory 512m nginx
```

限制 CPU：

```bash
docker run -d --name app --cpus 1.5 nginx
```

查看统计：

```bash
docker stats
```

如果容器因为内存不足被杀，常见状态是 OOMKilled。

Docker 查看：

```bash
docker inspect app
```

Kubernetes 查看：

```bash
kubectl describe pod pod-name
```

容器网络问题有时其实是资源问题：

- CPU 打满导致请求超时。
- 内存不足导致进程被杀。
- 文件描述符耗尽导致无法建立连接。
- 线程池耗尽导致端口还在监听但请求不响应。

### 16.11 Kubernetes 中的进程管理

Kubernetes 也以容器主进程为核心。

如果容器主进程退出，kubelet 会根据重启策略处理。

Pod 重启策略：

```yaml
spec:
  restartPolicy: Always
```

常见值：

- `Always`：总是重启，Deployment 默认。
- `OnFailure`：失败时重启，Job 常用。
- `Never`：不重启。

查看重启次数：

```bash
kubectl get pod
```

查看原因：

```bash
kubectl describe pod pod-name
kubectl logs pod-name
kubectl logs pod-name --previous
```

`--previous` 用于查看上一次崩溃容器的日志。

### 16.12 优雅终止

Kubernetes 删除 Pod 时，大致流程：

```text
Pod 标记为 Terminating
从 Service Endpoint 中移除
执行 preStop hook
发送 SIGTERM
等待 terminationGracePeriodSeconds
超时后 SIGKILL
```

示例：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: graceful-demo
spec:
  terminationGracePeriodSeconds: 30
  containers:
    - name: app
      image: my-app
      lifecycle:
        preStop:
          exec:
            command: ["sh", "-c", "sleep 5"]
```

注意：

- `preStop` 会占用优雅终止时间。
- 应用仍然需要处理 SIGTERM。
- 长连接服务要特别关注连接排空。

### 16.13 健康检查

Kubernetes 常见探针：

- `startupProbe`
- `readinessProbe`
- `livenessProbe`

示例：

```yaml
readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10

livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
```

含义：

- readinessProbe 失败：Pod 不接收 Service 流量。
- livenessProbe 失败：容器会被重启。
- startupProbe 用于启动慢的应用，避免过早被 livenessProbe 杀掉。

网络服务建议至少配置 readinessProbe。

否则应用进程虽然启动了，但端口还没准备好，Service 就可能把流量打进来。

### 16.14 进程与端口排查

当访问容器失败时，要确认进程真的在监听。

Docker：

```bash
docker exec -it web ss -lntp
```

Kubernetes：

```bash
kubectl exec -it pod-name -- ss -lntp
```

如果没有 `ss`，可以临时使用调试镜像，或查看应用日志。

常见监听错误：

```text
127.0.0.1:8080
```

这表示只监听本地回环地址。

更适合容器服务的监听方式：

```text
0.0.0.0:8080
```

监听地址区别：

- `127.0.0.1`：只允许同一网络命名空间内部访问。
- `0.0.0.0`：监听该命名空间内所有 IPv4 地址。
- 容器 IP：只监听指定容器网卡地址。

### 16.15 一个完整例子：为什么端口映射了仍然访问失败

启动容器：

```bash
docker run -d -p 8080:8080 my-app
```

外部访问失败：

```bash
curl http://宿主机IP:8080
```

排查：

```bash
docker ps
docker logs my-app
docker exec -it my-app ss -lntp
```

如果看到：

```text
LISTEN 0 128 127.0.0.1:8080
```

说明应用只监听了容器自己的本地回环。

虽然 Docker 已经把宿主机 `8080` 转发到容器 IP 的 `8080`，但容器内没有进程监听容器 IP 上的 `8080`。

修复方式：

```text
把应用监听地址从 127.0.0.1 改成 0.0.0.0
```

例如 Spring Boot：

```properties
server.address=0.0.0.0
server.port=8080
```

Node.js：

```javascript
app.listen(8080, "0.0.0.0")
```

Python Flask：

```python
app.run(host="0.0.0.0", port=8080)
```

### 16.16 进程管理基本原则

容器进程管理建议遵循这些原则：

1. 一个容器优先运行一个主进程。
2. 主进程以前台方式运行。
3. Dockerfile 优先使用 exec 格式。
4. 应用正确处理 SIGTERM。
5. 日志输出到 stdout/stderr。
6. 服务监听 `0.0.0.0`，不要只监听 `127.0.0.1`。
7. Kubernetes 中配置 readinessProbe。
8. 启动慢的应用使用 startupProbe。
9. 使用资源限制，并观察 OOMKilled、CPU 打满等问题。
10. 排查网络问题时，同时检查进程是否存活、是否监听、是否过载。

## 第十七章：常见故障排查

网络问题不要凭感觉猜，要按路径分层排查。

### 15.1 排查思路

从近到远：

1. 应用是否监听正确地址和端口。
2. 容器或 Pod 是否有 IP。
3. 路由是否正确。
4. DNS 是否正常。
5. 防火墙或网络策略是否阻断。
6. NAT 或端口映射是否正确。
7. 跨节点网络是否正常。
8. 外部负载均衡、安全组是否正常。

### 15.2 容器内服务无法访问

检查服务监听：

```bash
docker exec -it web sh
ss -lntp
```

如果应用只监听 `127.0.0.1`：

```text
127.0.0.1:8080
```

那么其他容器或外部无法通过容器 IP 访问。

应该监听：

```text
0.0.0.0:8080
```

这是非常常见的问题。

### 15.3 容器不能访问外网

检查 IP：

```bash
docker exec web ip addr
```

检查路由：

```bash
docker exec web ip route
```

测试 IP 连通：

```bash
docker exec web ping -c 3 8.8.8.8
```

测试 DNS：

```bash
docker exec web nslookup example.com
```

检查宿主机转发：

```bash
sysctl net.ipv4.ip_forward
```

应该为：

```text
net.ipv4.ip_forward = 1
```

临时开启：

```bash
sysctl -w net.ipv4.ip_forward=1
```

检查 NAT：

```bash
iptables -t nat -L -n -v
```

### 15.4 外部访问不到容器

检查容器是否运行：

```bash
docker ps
```

检查端口映射：

```bash
docker port web
```

检查宿主机监听：

```bash
ss -lntp | grep 8080
```

本机测试：

```bash
curl http://127.0.0.1:8080
```

局域网测试：

```bash
curl http://宿主机IP:8080
```

如果本机可以，外部不行，重点检查：

- 宿主机防火墙。
- 云服务器安全组。
- 端口绑定是否只绑定了 `127.0.0.1`。
- 上游负载均衡配置。

### 15.5 Kubernetes Service 访问失败

查看 Service：

```bash
kubectl get svc
kubectl describe svc web
```

查看 Pod label：

```bash
kubectl get pod --show-labels
```

查看 Endpoint：

```bash
kubectl get endpoints web
kubectl get endpointslice -l kubernetes.io/service-name=web
```

如果 Endpoint 为空，通常是：

- selector 写错。
- Pod label 不匹配。
- Pod 未 Ready。

检查端口：

```bash
kubectl describe svc web
kubectl describe pod pod-name
```

重点看：

- `port`
- `targetPort`
- 容器实际监听端口

### 15.6 Kubernetes DNS 失败

查看 CoreDNS：

```bash
kubectl -n kube-system get pod -l k8s-app=kube-dns
kubectl -n kube-system logs deploy/coredns
```

进入测试 Pod：

```bash
kubectl run dns-test --image=busybox:1.36 -it --rm --restart=Never -- sh
```

测试：

```bash
nslookup kubernetes.default
nslookup web.default.svc.cluster.local
```

如果 DNS 失败，检查：

- CoreDNS Pod 是否 Running。
- kube-dns Service 是否存在。
- Pod `/etc/resolv.conf` 是否正确。
- NetworkPolicy 是否阻止访问 DNS。

DNS 通常使用：

- UDP 53
- TCP 53

### 15.7 跨节点 Pod 不通

检查 Pod IP 和节点：

```bash
kubectl get pod -o wide
```

在源 Pod 中 ping 目标 Pod IP：

```bash
kubectl exec -it source-pod -- ping 目标PodIP
```

检查节点路由：

```bash
ip route
```

检查 CNI Pod：

```bash
kubectl -n kube-system get pod
```

根据插件查看日志：

```bash
kubectl -n kube-system logs daemonset/flannel
kubectl -n kube-system logs daemonset/calico-node
kubectl -n kube-system logs daemonset/cilium
```

重点检查：

- CNI 是否正常运行。
- 节点之间 UDP VXLAN 端口是否可达。
- BGP 邻居是否建立。
- MTU 是否匹配。
- 云安全组是否阻断节点间流量。

### 15.8 NetworkPolicy 导致不通

Kubernetes 默认没有 NetworkPolicy 时，Pod 之间通常全通。

一旦某个 Pod 被 NetworkPolicy 选中，访问规则就会变成白名单模型。

查看：

```bash
kubectl get networkpolicy
kubectl describe networkpolicy policy-name
```

常见问题：

- 只允许了业务端口，忘记允许 DNS。
- namespaceSelector 和 podSelector 写错。
- 以为策略会生效，但 CNI 不支持 NetworkPolicy。
- 以为策略不生效，其实 Pod 已经被某条策略选中。

## 第十八章：实验清单

这一章给出一组循序渐进的实验。建议在测试机上执行，不要直接在生产环境操作。

### 实验 1：观察 Docker 默认网络

```bash
docker network ls
ip addr show docker0
docker network inspect bridge
```

目标：

- 找到默认 bridge 网络。
- 找到 docker0 的 IP。
- 理解默认网段。

### 实验 2：启动容器并查看网络

```bash
docker run -d --name web nginx
docker inspect web
docker exec web ip addr
docker exec web ip route
```

目标：

- 找到容器 IP。
- 找到默认网关。
- 理解容器 eth0。

### 实验 3：测试容器访问外网

```bash
docker run --rm busybox ping -c 3 8.8.8.8
docker run --rm busybox nslookup example.com
```

目标：

- 区分 IP 连通问题和 DNS 问题。

### 实验 4：发布端口

```bash
docker run -d --name web-pub -p 8080:80 nginx
curl http://127.0.0.1:8080
docker port web-pub
```

目标：

- 理解 `宿主机端口:容器端口`。
- 理解 `EXPOSE` 与 `-p` 的区别。

### 实验 5：自定义 bridge 网络

```bash
docker network create demo-net
docker run -d --name web1 --network demo-net nginx
docker run -it --rm --name client --network demo-net busybox sh
```

在 `client` 中：

```bash
ping web1
wget -qO- http://web1
```

目标：

- 验证 Docker 内置 DNS。
- 理解同网络容器通信。

### 实验 6：一个容器连接两个网络

```bash
docker network create front
docker network create back

docker run -d --name proxy --network front nginx
docker run -d --name api --network back nginx

docker network connect back proxy
docker inspect proxy
```

目标：

- 理解多网卡容器。
- 理解前后端网络隔离。

### 实验 7：手动创建 namespace 和 veth

```bash
ip netns add ns1
ip link add veth-host type veth peer name veth-ns
ip link set veth-ns netns ns1

ip addr add 10.0.0.1/24 dev veth-host
ip link set veth-host up

ip netns exec ns1 ip addr add 10.0.0.2/24 dev veth-ns
ip netns exec ns1 ip link set veth-ns up
ip netns exec ns1 ip link set lo up

ping -c 3 10.0.0.2
ip netns exec ns1 ping -c 3 10.0.0.1
```

清理：

```bash
ip netns del ns1
```

目标：

- 从零理解容器网络的底层结构。

### 实验 8：Kubernetes Service 排查

创建 Deployment 和 Service：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 2
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx
          ports:
            - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
```

查看：

```bash
kubectl get pod -o wide
kubectl get svc web
kubectl get endpoints web
kubectl run test --image=busybox:1.36 -it --rm --restart=Never -- sh
```

在测试 Pod 中：

```bash
wget -qO- http://web
nslookup web
```

目标：

- 理解 Service DNS。
- 理解 Service 到 Pod 的后端映射。

## 第十九章：总结

容器网络可以从三层理解。

第一层是 Linux 基础：

- Network Namespace 让容器拥有独立网络栈。
- veth pair 像虚拟网线。
- Linux Bridge 像虚拟交换机。
- 路由表决定下一跳。
- iptables/nftables 实现 NAT、防火墙和转发。

第二层是 Docker 网络：

- bridge 是默认模型。
- user-defined bridge 更适合实际项目。
- host 共享宿主机网络，性能好但隔离弱。
- none 完全手工配置。
- container 共享另一个容器网络命名空间。
- overlay、macvlan、ipvlan 解决更特殊的网络需求。

第三层是 Kubernetes 网络：

- 每个 Pod 有自己的 IP。
- 同一个 Pod 内多个容器共享网络。
- Pod 之间默认应该可直接通信。
- CNI 插件负责实现 Pod 网络。
- Service 提供稳定入口。
- Ingress 提供 HTTP/HTTPS 入口。
- DNS 负责服务发现。
- NetworkPolicy 负责访问控制。

学习容器网络时，要记住一个核心方法：

> 永远沿着数据包路径排查。

例如外部访问容器失败，不要只看应用日志，而要按顺序检查：

```text
客户端
  -> DNS
  -> 外部负载均衡
  -> 宿主机端口
  -> NAT/Service/Ingress
  -> 容器或 Pod IP
  -> 应用监听端口
```

只要能把数据包从哪里来、到哪里去、经过哪些转发点讲清楚，容器网络就不再神秘。
